## 📱 Visão Geral do Projeto

**Catalyze** é um aplicativo iOS nativo para Engineering Managers (EMs) gerenciarem desenvolvimento de times de engenharia. É a versão mobile de um app web React/TypeScript.

### Tecnologias Principais
- **Linguagem**: Swift (100%)
- **UI Framework**: SwiftUI
- **Persistência**: SwiftData + CloudKit
- **Arquitetura**: Observable pattern (@Observable)
- **Plataforma**: iOS/iPadOS (suporte futuro para macOS)

### Funcionalidades Core
1. **Gestão de Time**: Perfis de membros com senioridade customizável
2. **Tracking de Skills**: Forças (Strengths) e Áreas de Crescimento (Weaknesses)
3. **Tech Stack**: Proficiência em tecnologias
4. **Observações**: Notas sobre interações e performance
5. **IDPs**: Individual Development Plans com ações rastreáveis
6. **Promotion Readiness**: Critérios e tracking de prontidão para promoção
7. **AI Insights**: Integração com Claude AI para análises e recomendações

## 🏗️ Arquitetura

### Padrão de Arquitetura
```
CatalyzeApp (Entry Point)
    ↓
AppLayout (NavigationSplitView)
    ├── Sidebar (Team / Insights / Settings)
    └── Detail Pane (View baseada em AppStore.activeView)
        ├── TeamView
        ├── MemberView
        ├── InsightsView
        └── SettingsView
```

### Camadas

#### 1. **App Entry** (`CatalyzeApp.swift`)
- Inicializa `ModelContainer` (SwiftData + CloudKit)
- Injeta `AppStore` no environment
- Configura `SeniorityService` para níveis customizáveis
- Gerencia fallback para storage local se CloudKit falhar

#### 2. **State Management** (`AppStore.swift`)
- State transiente da UI (navegação, seleção)
- Settings persistidos (API key via Keychain, profile via UserDefaults)
- Mutations que aceitam `ModelContext` para CRUD
- **NÃO** armazena arrays de dados - usa `@Query` nas views

#### 3. **Data Models** (SwiftData)

**Principais Models:**

```swift
@Model TeamMember {
    // Core
    - id, name, role, seniorityRaw, photoUrl, photoData
    
    // Relationships (cascade delete)
    - tags: [StrengthWeakness]
    - stack: [StackEntry]
    - observations: [TeamObservation]
    - idps: [DevelopmentPlan]
    - promotionRecords: [PromotionReadiness]
    - profileEvents: [ProfileEvent]
    
    // Mentoring (nullify delete)
    - mentor: TeamMember?
    - mentorName: String?
    - externalMenteesRaw: String?
}

@Model StrengthWeakness {
    - kindRaw: String (strength/weakness)
    - category: String (predefined ou custom)
    - intensityRaw: String (Emerging/Solid/Strong ou Emerging/Developing/Blocking)
    - note: String?
    - member: TeamMember?
}

@Model DevelopmentPlan {
    - title, objective, targetDate
    - statusRaw: String (Active/OnHold/Completed)
    - actions: [IDPAction] (cascade)
    - linkedGrowthAreaId: String? (ID de um StrengthWeakness)
}

@Model PromotionReadiness {
    - targetTierRaw: String
    - statusRaw: String (NotReady/InProgress/Ready)
    - criteria: [PromotionCriterion] (cascade)
    - aiAssessment: String?
}
```

**Seniority Customization:**

- Relationships sempre com inverse (CloudKit requirement)

final class MyModel {
    // 1. Properties com defaults (CloudKit requirement)
    var id: String = UUID().uuidString
    var name: String = ""
    var createdAt: Date = Date()
    
    // 2. Raw values para enums

## 🐛 Known Issues & Workarounds

### CloudKit Initialization
**Issue**: CloudKit pode falhar em simulador ou sem iCloud configurado.

**Workaround**: `makeContainer()` já tem fallback para local-only.

**Debug**: Use `makeLocalOnlyContainer()` para desabilitar CloudKit temporariamente.

### @Query em Previews
**Issue**: @Query pode crashar em previews sem container adequado.

**Workaround**: Sempre use `.modelContainer(try! PersistenceController.makePreviewContainer())`.

### Color Dark Mode
**Issue**: Cores custom podem não adaptar bem ao dark mode.

**Workaround**: Use `Color(.systemBackground)` ou compute cores dinamicamente:
```swift
@Environment(\.colorScheme) var colorScheme
var adaptiveColor: Color {
    colorScheme == .dark ? .white : .black
}
```
